# HT-014  Task Prioritization

## What is it?
A post-processing step that scores and reorders the AI-generated tasks using deterministic rules  ensuring the final task list is capped at 7 items and weighted toward what matters most.

## Problem Statement
The LLM may return 10 tasks with inconsistent priorities. The AI's own priority assignments are based on context, but certain objective factors  overdue status, goal importance, project neglect  should always outweigh the LLM's subjective ordering.

## Why Do We Need It?
- The LLM is good at generating relevant tasks but not always reliable at ranking them
- Deterministic rules give the team control over planning behaviour without retraining the model
- Capping at 7 tasks prevents overwhelming the user

## Objective / Goal
A `prioritize_tasks(tasks, context)` function that scores each task against a defined ruleset and returns the top 7 in descending priority order.

## Scope
### In Scope
- Scoring each task using additive rules based on context data
- Sorting tasks by total score descending
- Capping the result at 7 tasks
- Logging the score breakdown for debugging

### Out of Scope
- User-defined priority rules or preferences (future phase)
- Machine learning-based prioritisation
- Prioritising tasks outside of the daily planning flow

## User Stories
- As a user, I want overdue tasks to always appear near the top of my plan so I do not miss deadlines
- As a user, I want tasks linked to high-priority goals to be ranked higher so my most important work gets attention first
- As a developer, I want the scoring logic to be transparent and logged so I can debug unexpected ordering

## Requirements
- REQ-1: Each task receives a numeric score starting at 0
- REQ-2: Score rules (additive):
  - +3 if linked to a goal with `priority = 1` (high)
  - +2 if the task is overdue (has a `due_date` before today)
  - +1 if linked to a project with `progress < 30%` (neglected project)
  - -1 if a task with a very similar title was completed in the last 2 days (recently done)
- REQ-3: Tasks are sorted by score descending
- REQ-4: The output list is capped at 7 tasks  any beyond that are dropped
- REQ-5: Each task in the output must carry its score in a `priority_score` field for the frontend to optionally display
- REQ-6: Score breakdown per task must be logged at DEBUG level

## Solution Overview
- Create `backend/services/prioritization_service.py` with a single public function `prioritize_tasks(tasks, context)`
- The function accepts the list of `GeneratedTask` objects and the `UserContext` from HT-011
- For each task: look up its linked goal's priority from the context, check if it is overdue, check if its linked project has low progress, check if a similar task was recently completed
- Assign scores, sort, cap at 7, return
- The "recently completed" check uses a simple string similarity check on task titles (not semantic search)  if the first 5 words match a task completed in the last 2 days, apply the -1

## Acceptance Criteria
- [ ] A task linked to a high-priority goal scores at least 3 points higher than an unlinked task
- [ ] An overdue task always appears above a non-overdue task of the same base priority
- [ ] The output never contains more than 7 tasks even if 15 were generated by the LLM
- [ ] Each returned task has a `priority_score` field with a numeric value
- [ ] Score breakdown is visible in server logs at DEBUG level

## Success Metrics
- The top task in the plan is overdue or linked to a high-priority goal in at least 80% of test scenarios
- Users do not report feeling that the task ordering is random or counterintuitive

## Risks / Dependencies
- **Dependency:** HT-013 must produce the list of generated tasks that this function processes
- **Dependency:** HT-011 context is passed directly  the `active_goals` and `active_projects` fields must include `priority` and `progress` values
- **Risk:** The -1 "recently done" rule could suppress important recurring tasks  monitor user feedback and be ready to remove or adjust it
